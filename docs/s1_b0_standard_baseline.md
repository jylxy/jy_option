# S1 B0 标准基准定义

## 1. 为什么需要 B0

B0 是 S1 卖权策略的“干净基准”，不是优化版本。它的作用是回答一个最基础的问题：

```text
如果整个产品只做 S1 卖权，使用全品种、低 delta、次近到期、50% 目标保证金、真实费用和真实保证金率，
不做 IV/RV/降波/趋势/组合择时过滤，朴素卖权本身到底能贡献多少收益、承担多少回撤？
```

后续所有 S1 改进都应该先和 B0 比，而不是只和上一个复杂版本比。这样我们能分清楚：收益来自卖权底层风险溢价，还是来自某个过滤、评分、择时或仓位模块。

## 2. B0 的策略定位

B0 只表达“全品种系统性收权利金”的基准，不表达乐得式择时，也不表达我们的最终策略形态。

它应该足够朴素，但不能因为实现口径扭曲结果。因此本版 B0 固定以下原则：

- 不做 IV/RV、降波、forward vega、趋势动量、板块集中度、相关性、stress score 等智能过滤。
- 使用期货公司/交易所口径的手续费和保证金率。
- 费用、滑点、成交量限制、到期虚值归零和止损防跳变都保留，避免基准过度乐观。
- 目标保证金使用率固定为 50%，S1 本身也最多使用 50% 资金。

## 3. 交易品种池

默认扫描当前系统可读到的全部期权品种，包括商品期权、ETF 期权和股指期权。

品种上市后不需要人工加入白名单。只要当天存在有效期权链、真实标的价格、合约乘数、手续费和保证金参数，就进入每日扫描。实际能否开仓由候选合约是否存在、整数手数、保证金约束、成交量限制和费用过滤共同决定。

## 4. 到期选择

B0 使用“次近到期”，代码口径为：

```text
s1_expiry_mode = nth_expiry
s1_expiry_rank = 2
```

这里的“次近到期”不是自然月份的“下个月”，而是当天有效到期序列中按 DTE 从小到大排序后的第 2 个到期日。

这样处理更适合商品期权，因为商品期权合约月份并不总是严格对应自然次月。如果当天只有 1 个有效到期日，则该品种当天不开仓。

## 5. 选腿规则

每天对每个品种的次近到期同时扫描 Put 和 Call：

- Put：只卖 OTM Put。
- Call：只卖 OTM Call。
- Delta：`abs(delta) <= 0.10`。
- Delta 下限：不设硬下限。
- 排序：优先选择最接近 `abs(delta)=0.10` 的合约，再用流动性和合约代码做稳定排序。
- 每个品种、每个方向、每个扫描日最多选择 5 个卖方合约。
- 不买保护腿，不做价差，只记录裸卖基准。

这版 B0 不再“所有符合条件的深虚值合约都卖”。原因是全卖会把品种结构变成交易所挂牌密度测试，而不是策略基准；每侧 5 个低 delta 合约更接近可解释的执行价梯队。

## 6. P/C 结构

B0 不做趋势判断，也不主动偏 Put 或偏 Call。每个品种每天默认尝试双侧卖权：

```text
每个品种预算 = 当日可用 S1 预算 / 当日可交易品种数
Put 侧预算 = 每个品种预算 / 2
Call 侧预算 = 每个品种预算 / 2
```

实际结果中 P/C 仍可能偏离 1:1，这是正常的，主要来自：

- 某一侧没有满足 `abs(delta)<=0.10` 且 OTM 的合约。
- 某一侧权利金不足以覆盖往返手续费。
- 整数手数和保证金不同。
- 该侧触发止损、到期或成交量限制。
- 某些 ETF/股指/商品品种在某段时间的数据可用性不同。

因此 B0 的目标不是强制 P/C 手数完全相等，而是保证开仓逻辑本身不主观偏向某一侧。

## 7. 仓位和资金分配

核心仓位参数：

```text
capital = 10,000,000
margin_cap = 0.50
s1_margin_cap = 0.50
margin_per = 0.02
s1_baseline_equal_weight_products = true
s1_baseline_equal_weight_contracts = true
s1_baseline_max_contracts_per_side = 5
```

解释：

- 组合总保证金使用率最高 50%。
- S1 自身保证金使用率最高 50%。
- 每日按当日可交易品种数等权分配预算。
- 每个品种内部 P/C 各拿一半预算。
- 每侧最多 5 个合约，合约之间按剩余预算等权切分。
- 同一品种、同一方向、同一到期允许在后续交易日继续加仓，直到组合和品种预算自然限制住。

实际保证金使用率可能低于 50%。这不一定是 bug，常见原因是合格候选不足、整数手数向下取整、单合约保证金太大、开仓成交量限制、费用过滤或已有仓位浮亏导致 NAV 变化。

## 8. 成本、成交和流动性

B0 保留实盘必要成本假设：

```text
option_fee_use_broker_table = true
margin_ratio_use_broker_table = true
s1_min_premium_fee_multiple = 1.0
volume_limit_pct = 1.0
execution_slippage_enabled = true
execution_open_slippage_pct = 0.002
execution_close_slippage_pct = 0.002
execution_stop_slippage_pct = 0.005
execution_slippage_apply_to_expiry = false
```

其中 `s1_min_premium_fee_multiple=1.0` 表示单张合约权利金现金额至少要覆盖一开一平的往返手续费。否则即使 delta 合格，也不值得做。

成交仍使用当前系统里的 T+1 全天 VWAP 口径，并保留成交量限制。这个设定不是为了让 B0 变聪明，而是为了避免用不可成交价格构造收益。

## 9. 退出规则

B0 默认持有到期，不止盈、不主动滚仓、不做临近到期提前处理。

唯一提前退出是权利金硬止损：

```text
premium_stop_multiple = 2.5
premium_stop_requires_daily_iv_non_decrease = false
intraday_stop_confirmation_enabled = true
```

也就是说，期权价格达到开仓权利金的 2.5 倍时触发止损，但仍使用当前盘中止损防跳变确认逻辑，避免因为深虚值小成交量合约的瞬时异常报价直接打出假止损。

到期处理按真实到期内在价值结算：

- 虚值到期：期权价值归零，卖方保留权利金。
- 实值到期：按内在价值结算卖方亏损。

## 10. B0 明确不包含什么

为了保持基准干净，B0 不包含：

- IV 预热和 IV 分位过滤。
- IV/RV carry 过滤。
- 降波确认和 forward vega 过滤。
- 趋势动量下的 P/C 偏移。
- 板块集中度、相关性和 cash Greeks 组合控制。
- stress score 候选评分。
- stress budget sizing。
- 止盈。
- 止损后冷却期和重开规则。
- 保护腿或比例价差。

这些都属于后续 S1 改进版，而不是 B0。

## 11. 当前配置文件

标准 B0 配置文件为：

```text
config_s1_baseline_b0_all_products_stop25.json
```

建议正式回测使用新标签，避免和旧口径 B0 混在一起：

```text
s1_b0_standard_stop25_allprod_2022_latest
```

旧标签 `s1_baseline_b0_allprod_2022_latest` 对应的是标准化前的运行结果，不应再作为最终 B0 结论。

## 12. B0 结果必须检查的诊断

每次 B0 长回测完成后，至少检查：

- 年化收益、最大回撤、Sharpe、Calmar。
- 年度、季度、月份收益，尤其是 2025 年 4 月和 7 月升波期。
- 平均和最高保证金使用率，确认是否接近 50%。
- 平均活跃品种数、活跃合约数，以及活跃品种是否随时间异常减少。
- P/C 手数、P/C 权利金、P/C 保证金和 P/C PnL。
- ETF、股指、商品三类市场分别贡献。
- 止损次数、止损品种、止损成交价相对 2.5x 阈值的偏离。
- 到期虚值归零次数、到期实值次数、到期实值亏损。
- 权利金收入、手续费、滑点、到期损益、止损损益拆解。
- Greek 归因中的 theta、vega、delta、gamma、residual。

如果 B0 收益很低，优先检查：

- 实际保证金是否长期没有打满 50%。
- 是否大量候选被费用过滤或成交量限制过滤。
- 次近到期是否在部分商品上候选不足。
- ETF/股指是否因为数据映射、手续费、乘数或标的价格缺失没有进入样本。
- P/C 偏离是否来自真实候选不足，而不是代码只卖单侧。
